% ABSTRACT

\begin{abstract}
Wi-Fi deauthentication attacks force clients off a network by sending forged management frames that the 802.11 standard requires clients to obey. The attack is trivial to execute, enables WPA handshake capture for offline cracking, and remains effective on any network without Protected Management Frames. Existing detection work either targets enterprise hardware, uses synthetic traffic for evaluation, or tests only a single detection method without comparison. We implement two detection approaches---threshold-based counting and statistical z-score anomaly detection---on a Raspberry Pi Zero W and an ESP32 microcontroller, then test both against a real Pwnagotchi attacker across five scenarios ranging from aggressive floods to slow evasion attempts. We measure detection rate, false positive rate, detection latency, resource consumption, and the minimum attack rate each approach can still catch. \placeholder{1-2 sentences summarizing the key finding once experiments are done.} Both implementations are released as open source.
\end{abstract}

% I. INTRODUCTION
% ============================================================
\section{Introduction}

Modern Wi-Fi networks encrypt data traffic with WPA2 or WPA3, but management frames---the frames responsible for connecting and disconnecting clients---are sent in the clear on any network that has not enabled Protected Management Frames (PMF/802.11w)~\cite{ieee80211w}. A deauthentication attack exploits this gap: the attacker spoofs the access point's MAC address and sends a deauthentication frame to the victim, which the victim's adapter treats as legitimate and disconnects. Tools like Pwnagotchi~\cite{pwnagotchi} automate this entirely, sending targeted deauths to force reconnections and capture WPA handshakes for offline cracking.

The attack requires nothing more than a \$10 single-board computer with a Wi-Fi chipset that supports monitor mode. It works against essentially all WPA2 networks without PMF, which still represents the majority of home and small-business deployments. WPA3 mandates PMF, but adoption is slow---legacy devices do not support it, and many routers disable it by default for compatibility.

On the detection side, there is a gap between what exists and what is needed. Enterprise wireless intrusion detection systems (Cisco Adaptive WIPS, Aruba RFProtect) can flag deauth floods, but they cost thousands of dollars and assume centralized infrastructure. Hobbyist projects like the SpacehuhnTech DeauthDetector~\cite{spacehuhn_detector} prove that a \$5 ESP8266 can count deauth frames and light an LED, but none of them have been evaluated in any rigorous way---no false positive rates, no latency measurements, no evasion testing.

The academic literature is thin. The closest works test on synthetic traffic~\cite{nature2025_deauth}, evaluate a single method with minimal metrics~\cite{ijert2024_deauth}, or run ML classifiers on laptop hardware that cannot be deployed on constrained devices~\cite{ieee2023_deauth_ml}. Nobody has compared detection approaches head-to-head on the same constrained platform against a real attacker, and nobody has systematically tested what happens when the attacker slows down to evade detection.

This paper addresses that gap with three contributions:

\begin{enumerate}
    \item A comparative evaluation of threshold-based and statistical (z-score) detection on two platforms (Raspberry Pi Zero W at \$15 and ESP32 at \$5), tested against a real Pwnagotchi attacker rather than synthetic traffic.
    \item A systematic evasion analysis that identifies the minimum attack rate each approach can detect, characterizing where simple detection methods break down.
    \item Open-source implementations for both platforms with full experimental data for reproducibility.\footnote{\url{https://github.com/BushraKhattabAlawneh/wifi-deauth-detection}}
\end{enumerate}

\section{Background}
\label{sec:background}

\subsection{802.11 Frame Types and the Management Frame Problem}

The 802.11 protocol uses three frame categories: data frames carry user payload, control frames handle medium access (RTS/CTS, ACK), and management frames handle connection state---beacons, probe requests, authentication, association, and deauthentication. The Frame Control field encodes the type and subtype: subtype 0x0C is deauthentication, 0x0A is disassociation.

The key problem is that management frames carry no authentication or integrity protection in pre-802.11w networks. Any radio within range can craft and transmit them. The receiving station has no way to distinguish a legitimate deauthentication frame from the AP (e.g., due to administrative policy) from a forged one sent by an attacker three meters away.

\subsection{How Deauthentication Attacks Work}

The attacker puts a wireless adapter into monitor mode, scans channels to find active APs and their associated clients, then sends deauthentication frames with the source address set to the target AP's BSSID. The frame carries a reason code (commonly reason 7, ``Class 3 frame received from non-associated station'') that the client firmware obeys immediately.

The attack works because:
\begin{itemize}
    \item Management frames require no prior authentication to send
    \item The standard mandates that clients act on deauthentication frames immediately
    \item MAC address spoofing is trivial---the attacker copies the AP's BSSID into the source field
    \item Continuous deauthentication prevents the client from staying connected
\end{itemize}

This is not just a denial-of-service problem. When a deauthenticated client reconnects, it performs a fresh 4-way handshake. If the attacker captures this handshake, they can perform an offline dictionary attack against the network password. Tools like Pwnagotchi are built specifically around this workflow.

\subsection{Protected Management Frames and Their Limitations}

IEEE 802.11w~\cite{ieee80211w}, ratified in 2009, adds cryptographic protection to management frames using keys derived from the existing security association. Unicast management frames are encrypted; broadcast ones carry a Message Integrity Check. WPA3 requires PMF.

In practice, PMF deployment remains limited. Many IoT devices, older laptops, and smartphones do not support it. Routers operating in WPA2/WPA3 transitional mode often disable PMF to maintain compatibility with legacy clients. Even networks with PMF enabled are not fully immune---an attacker can still send broadcast deauthentication frames, though PMF-capable clients will reject them. For the foreseeable future, passive detection remains a necessary defense layer.

\subsection{Pwnagotchi as a Test Attacker}

Pwnagotchi~\cite{pwnagotchi} runs on a Raspberry Pi Zero W and uses reinforcement learning to optimize its handshake capture strategy. Unlike \texttt{aireplay-ng}, which sends a fixed number of deauth frames at a constant rate, Pwnagotchi varies its timing, hops channels, and selects targets based on past success. This semi-random behavior makes it more representative of a real attacker than scripted tools, which is why we use it as our primary test source.

\section{Related Work}
\label{sec:related}

\subsection{Academic Work on Deauth Detection}

We found four directly relevant academic studies. Each addresses part of the problem, but none covers the full picture.

A 2025 study in Nature Scientific Reports~\cite{nature2025_deauth} deploys a CNN-LSTM hybrid model on a NodeMCU (ESP8266). The model achieves high classification accuracy, but the evaluation uses synthetic deauthentication traffic---generated frames injected into a capture file rather than a live attack. There is no detection latency measurement, no resource usage reporting, and no analysis of what happens when the attacker reduces intensity.

An IJERT 2024 paper~\cite{ijert2024_deauth} implements frame-count threshold detection on an ESP8266. It shows that on-device detection is feasible, but the evaluation consists of a single test with the attack running at full speed. There are no false positive experiments, no comparison with other approaches, and the threshold value appears chosen arbitrarily without sensitivity analysis.

A 2023 IEEE paper~\cite{ieee2023_deauth_ml} applies Random Forest and SVM classifiers to captured Wi-Fi traffic on a laptop. Detection accuracy is high, but the models require far more memory and compute than a microcontroller offers. The paper does not discuss constrained deployment at all.

A recent arXiv preprint~\cite{arxiv2025_deauth_testbed} presents a software-defined testbed for measuring network-level resilience to deauthentication. It characterizes the attack from the network's perspective rather than building an on-device detector, making it complementary to our work.

\subsection{Open-Source Detection Tools}

The hobbyist community has built several deauthentication detectors:

\begin{itemize}
    \item \textbf{SpacehuhnTech DeauthDetector}~\cite{spacehuhn_detector}: counts deauth frames per second on an ESP8266, lights an LED above a fixed threshold. No logging, no tuning guidance, no evaluation published.
    \item \textbf{Deauthalyzer}: a Python script for Linux that sniffs and displays deauth information. No detection logic beyond basic counting.
    \item \textbf{BriarIDS}: a Raspberry Pi home IDS covering many attack types. Not specialized for deauthentication, no published evaluation.
\end{itemize}

These projects confirm demand for low-cost detection but provide no data on how well their approaches actually work, when they fail, or how they compare.

\section{System Design}
\label{sec:design}

\subsection{Novelty of Approach}

Our contribution is not in the detection algorithms themselves---threshold counting and z-score anomaly detection are well-established techniques. The novelty lies in three aspects of how we apply and evaluate them:

First, \textit{comparative evaluation on the same constrained platform}. Prior work implements one method and reports its accuracy in isolation. Without a second method tested under identical conditions, there is no way to assess whether the chosen approach is actually better than simpler alternatives. We run both detectors on the same hardware against the same traffic, producing a direct comparison.

Second, \textit{testing against a real autonomous attacker}. Pwnagotchi's behavior---channel hopping, target selection, variable timing---produces traffic patterns that synthetic generators cannot replicate. An attacker that sends 100 deauths in one burst, waits 30 seconds, then sends 50 to a different client is qualitatively different from a constant-rate flood, and detection systems that only see the latter during evaluation may fail against the former.

Third, \textit{systematic evasion characterization}. Reporting that a detector works at full attack intensity is easy. The harder and more useful question is: at what point does it stop working? By sweeping attack rates from 20 down to 0.5 frames per second, we identify the exact boundary where each approach transitions from reliable detection to silence. This gives practitioners actionable guidance on what these systems can and cannot do.

\subsection{Architecture}

The detector runs as a passive sniffer. A wireless interface in monitor mode (RPi) or promiscuous mode (ESP32) captures all 802.11 frames on the configured channel. A filter stage passes only deauthentication (subtype 0x0C) and disassociation (subtype 0x0A) frames. For each matching frame, the system extracts the source MAC, destination MAC, BSSID, reason code, and timestamp, then feeds this to one or both detection engines. Both engines operate over a sliding time window and produce a binary alert signal. All frames and alerts are logged to CSV for offline analysis.

\subsection{Threshold Detector}

The simplest approach: count deauthentication frames in a sliding window and alert if the count crosses a fixed value.

Given a window size $W$ seconds and threshold $T$ frames:
\begin{equation}
\text{alert}(t) =
\begin{cases}
1 & \text{if } \sum_{i=t-W}^{t} f_i \geq T \\
0 & \text{otherwise}
\end{cases}
\end{equation}

We use $W = 5$\,s and $T = 10$ frames. A legitimate AP might send 1--2 deauths in a 5-second span when a client roams or times out; an attacker sends dozens to hundreds per second.

The implementation stores frame timestamps in a deque, evicts entries older than $W$ on each arrival, and checks the length. Memory usage is negligible (at most $T$ timestamps), and there is no warm-up---it alerts from the first frame if the rate is high enough.

The downside is rigidity. The threshold does not adapt: a network with frequent client roaming might generate enough legitimate deauths to cause false positives, while a slow attacker sending just below $T/W$ frames per second will never trigger it.

\subsection{Statistical Detector (Z-Score)}

This approach learns what ``normal'' looks like and flags deviations. It operates in two stages:

\textbf{Baseline collection:} For the first $N$ time windows (each $W$ seconds long), the detector records the deauth count per window without alerting. This builds a picture of the normal deauthentication rate for the specific environment.

\textbf{Anomaly detection:} After the baseline is established, each new window's count is compared to the baseline via z-score:
\begin{equation}
z = \frac{c - \mu}{\sigma}
\end{equation}
where $c$ is the current window's deauth count, $\mu$ and $\sigma$ are the mean and standard deviation of the baseline. An alert fires when $z \geq 3.0$. The baseline is maintained as a rolling buffer of the most recent $N$ window counts, so it gradually adapts if the environment changes.

We use $W = 5$\,s, $N = 20$ windows (100 seconds of warm-up), and $z_\text{threshold} = 3.0$. When $\sigma = 0$ (no variance in the baseline, i.e., consistently zero deauths), any count above $\mu + 1$ triggers an alert.

The advantage over threshold detection is adaptability---a noisy environment with occasional legitimate deauths will have a higher $\mu$ and $\sigma$, raising the bar for what counts as anomalous. The disadvantage is the warm-up blind spot: attacks during the first 100 seconds go undetected. There is also a subtle vulnerability to slow-ramp attacks that gradually increase intensity and poison the rolling baseline.

\subsection{Detection Algorithm}

Algorithm~\ref{alg:detection} shows the combined detection procedure. Each incoming frame is checked against both engines in sequence; the CSV log records per-frame state regardless of whether an alert fires.

\begin{algorithm}[htbp]
\caption{Combined threshold and statistical detection.}
\label{alg:detection}
\begin{algorithmic}[1]
\STATE \textbf{Input:} stream of 802.11 frames on monitored channel
\STATE \textbf{Parameters:} $W$ (window), $T$ (threshold), $N$ (baseline windows), $z_\text{thr}$ (z-score threshold)
\STATE Init: deque $Q \leftarrow \emptyset$, buffer $B \leftarrow \emptyset$, counter $c \leftarrow 0$, $t_w \leftarrow$ now
\FOR{each frame $f$}
    \IF{$f$.subtype $\notin \{$0x0C, 0x0A$\}$}
        \STATE \textbf{continue}
    \ENDIF
    \STATE $t \leftarrow$ timestamp of $f$
    \STATE \textit{// Threshold engine}
    \STATE Append $t$ to $Q$
    \WHILE{$Q$.front $< t - W$}
        \STATE Remove front of $Q$
    \ENDWHILE
    \STATE $\text{alert}_\text{thr} \leftarrow (|Q| \geq T)$
    \STATE \textit{// Statistical engine}
    \STATE $c \leftarrow c + 1$
    \IF{$t - t_w \geq W$}
        \STATE Append $c$ to $B$ (rolling, max $N$); $c \leftarrow 0$; $t_w \leftarrow t$
    \ENDIF
    \IF{$|B| \geq N$}
        \STATE $\mu \leftarrow \text{mean}(B)$; $\sigma \leftarrow \text{std}(B)$
        \IF{$\sigma = 0$}
            \STATE $\text{alert}_z \leftarrow (c > \mu + 1)$
        \ELSE
            \STATE $\text{alert}_z \leftarrow ((c - \mu)/\sigma \geq z_\text{thr})$
        \ENDIF
    \ELSE
        \STATE $\text{alert}_z \leftarrow$ false
    \ENDIF
    \STATE Log $(t,\; f.\text{src},\; f.\text{dst},\; f.\text{reason},\; \text{alert}_\text{thr},\; \text{alert}_z)$
\ENDFOR
\end{algorithmic}
\end{algorithm}

The algorithm runs in $O(1)$ amortized time per frame for both engines. The threshold engine's deque operations are bounded by the eviction loop (at most $T$ entries at any time). The statistical engine performs a fixed-size mean and standard deviation over $N$ values only at window boundaries, not per frame. Peak memory on the ESP32 is $T \times 4$ bytes (timestamps in the deque) plus $N \times 4$ bytes (baseline buffer)---under 200 bytes with our parameters.

\subsection{Design Decisions}

Several design choices deserve justification:

\textbf{Window size (5\,s):} Short enough to detect burst attacks within a few seconds, long enough to avoid triggering on isolated legitimate deauths. A 1-second window would alert on even a single burst of 10 frames (common during AP-initiated disconnects); a 10-second window would delay detection unacceptably.

\textbf{Threshold value (10 frames):} Based on observed legitimate behavior---an AP performing a clean disconnection sends 1--3 deauth frames to a single client. A margin of $3\times$ over the worst observed legitimate case gives 10.

\textbf{Baseline length (20 windows / 100\,s):} Long enough to capture the variance of a typical home network (occasional roaming events, background scans) but short enough that the detector becomes operational within two minutes of power-on.

\textbf{Z-score threshold (3.0):} Corresponds to $>$99.7\% confidence under a normal distribution. In practice, deauth counts per window are not normally distributed (they are zero-inflated), but z = 3.0 provides a reasonable balance between sensitivity and false positive suppression.

\subsection{Platform Implementations}

On the \textbf{Raspberry Pi}, the detector is written in Python using Scapy for packet capture. The USB Wi-Fi adapter is placed in monitor mode via \texttt{airmon-ng}. Both detection engines run on every captured frame, and results are written to a CSV file with per-frame granularity (timestamp, MACs, reason code, alert status per engine).

On the \textbf{ESP32}, the detector is written in C using the Espressif IDF promiscuous mode API. A callback registered with \texttt{esp\_wifi\_set\_promiscuous\_rx\_cb()} receives every frame; the callback checks the subtype byte and increments a counter. The main loop checks the window boundary every 100\,ms and outputs one CSV row per window over serial.

The detection logic is identical across platforms. Differences in measured performance come from the language runtime overhead (Python/Scapy vs. bare-metal C) and the capture mechanism (libpcap wrapper vs. hardware callback).

\subsection{Evasion Sweep}

To find the exact point where detection fails, we run a series of 60-second trials at controlled rates: 20, 15, 10, 8, 5, 3, 2, 1, and 0.5 frames per second. For each rate, we record whether each detector alerts and, if so, the time to first alert. This maps out the evasion boundary for both approaches.

\subsection{Metrics}

For each combination of detector, platform, and scenario, we report:

\begin{enumerate}
    \item \textbf{Detection Rate:} proportion of attack windows where the detector correctly fired an alert.
    \item \textbf{False Positive Rate:} proportion of non-attack windows where the detector incorrectly fired.
    \item \textbf{Detection Latency:} seconds between the first deauth frame arriving and the first alert.
    \item \textbf{Evasion Threshold:} the lowest attack rate (frames/sec) that still triggers detection.
    \item \textbf{Resource Usage:} CPU utilization and memory consumption during detection, measured with \texttt{psutil} on the RPi and heap reporting on the ESP32.
\end{enumerate}

\subsection{Deployment Considerations}

A detector in isolation is useful for awareness but not for response. Practical deployment should consider:
\begin{itemize}
    \item \textbf{Alerting:} the ESP32 can drive an LED or buzzer directly; the RPi can send network notifications (email, push, MQTT)
    \item \textbf{Channel coverage:} a single-channel sniffer misses attacks on other channels. Options include running multiple detectors (one per channel) or implementing channel hopping with per-channel state
    \item \textbf{Placement:} the detector must be within radio range of both the AP and potential attackers, which in practice means near the AP
    \item \textbf{Power:} the ESP32 can run for weeks on a USB battery pack; the RPi needs continuous power
\end{itemize}

\subsection{Limitations}

This work has several limitations that scope its conclusions:

\begin{enumerate}
    \item We monitor one channel at a time. An attacker on a different channel will not be detected.
    \item We test against one AP in one indoor environment. Multi-AP environments, outdoor deployments, and high-density settings may produce different baseline patterns.
    \item Our primary attacker is a Pwnagotchi. Other tools (ESP8266 Deauther, custom scripts with different timing profiles) might behave differently.
    \item We do not implement ML-based detection. While a lightweight decision tree could improve evasion resistance, it adds complexity that we defer to future work.
    \item We do not measure power consumption directly, only CPU and RAM.
\end{enumerate}

This motivates future work on feature-rich detection that considers not just frame counts but target diversity, reason code patterns, and timing regularity.

\subsection{Future Work}

\begin{itemize}
    \item \textbf{Adaptive thresholds with reinforcement learning:} an RL agent could tune detection parameters in real time based on environmental feedback, potentially combining the immediacy of threshold detection with the adaptability of statistical approaches.
    \item \textbf{Feature-based detection:} incorporating per-window features beyond frame count---target MAC diversity, reason code entropy, burst duration, sequence number gaps---could catch slow attacks that rate-based methods miss.
    \item \textbf{Multi-channel scanning:} rapid channel hopping with independent per-channel state to provide coverage across the full 2.4\,GHz band.
    \item \textbf{Power measurement:} direct current measurement to characterize battery life for unattended ESP32 deployment.
\end{itemize}
